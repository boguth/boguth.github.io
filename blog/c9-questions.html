May 15th - Asking the Right Questions

I asked this question of the Google+ DBC community about three weeks ago when I was first learning JavaScript:

Hey everyone: Can someone please explain this to me:

assert(
(typeof secretNumber === 'number'),
"The value of secretNumber should be a number.",
"1. "
)

How does javascript not see 'number' as a string? How does it know it means number?ï»¿

Through asking this question, and reading stack overflow's guide on how to ask good questions, I have learned that asking a question well takes forethought. To make this question, and any other question I may ask in the future, better, there are a few things that I could do.

First, I could have started by more thoroughly describing how I arrived at the problem. For example:

"I was working through a ruby challenge and was using RSPEC to test my solution. I got the first test to pass, but I wondered how."
Another thing I could have done was framed the problem more specifically. Instead of asking "How does Javascript not see 'number' as a string," I should have asked:
"If you observe the code below, you'll notice that it evalutes secretNumber against 'number.' I don't understand how this can return true when secretNumber is an integer and 'number' is a string."
Then I could have pasted the code below the previous paragraph.

Finally, another thing I could have done is researched the question online by predicting what more information would be asked of me. For example, if someone were to ask "are you using Javascript," that would have led me to research how Javascript evalutes types. I would have discovered that when javascript compared types, it converts those types to strings. So when it sees a number, the type is not a reserved word number, it's 'number.'

Essentially, asking a good question requires anticipating what the audience will need to answer it, doing your homework in terms of researching all aspects of the problem, and then framing the question with enough background and context to eliminate any ambiguity.
