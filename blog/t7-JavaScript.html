May 2nd, 2015 - JavaScript vs. Ruby

Coming from ruby, my experience with JavaScript has been one marked by giant moodswings, from frustration to "a-ha" moments. What I have discoverd, so far, is that depsite the different syntax, the two languages are very similar. In fact, a lot of the same types of fundamental capabilities exist in both languages. But, where Ruby is well defined, organized, and buttoned up, JavaScript is loosey goosey and kind of weird.

Take, for example, classes. In Ruby, classes define a set of behaviors and all objects of a given class share those behaviors. JavaScript doesn't really have classes in such a well defined sense. It has types, which describe kinds of data (string, boolean, integer, etc.).

Where things get confusing is when you want to start making JavaScript "classes" of your own. You would think that if types are the equivalent of a Ruby class (sorta), you would simply create your own types. Not so. Types describe data, as we said. There can't be any more or different kinds of data than the one's already defined. So, then, how do we create our own groupings of properties and behaviors?

The answer is with constructor functions. Constructor functions are defined just like any other function. The only difference is that you use the word "new" in front of a constructor function when you are assigning it to a variable. Make sense? No? Okay, let's water it down.

This is a function:
function example(){
}
When we want to call it or use it, we'll either type:
example();
Or we'll assign it to a variable and call the variable, like this.
var ex = function example(){
}
ex();// calling the fuction with a variable.

Constructor function are constructed the exact same way. Except, if you use the "new" keyword instead of the word function, like this:
var ex = new example();
some different things happen. To be clear, the only reason our example function is now a constructor function is because we said so. The only reason it's any different than a regular function is because when we are assigning it to a variable, we use the word "new" instead of "function."

"New" is a special thing. It does three primary things.

1. It creates a new object.

The variable we used to assign new exmaple(); doesn't point to our constructor function. It is an object in and of itself.

2. It sets a very special and usually hidden property called the Constructor property to the function object

What this means is that if we type ex instanceof example then that will return true.

3. It makes it so that all the objects constructed from the constructor function inherit its properties.

Here's where things get a bit more familiar. Just like class inheritence in Ruby, the "new" word makes all the objects created from our constructor inherit its properties. And, just like in ruby, was those properties are copied over, they are free to be used by the new object at will. Any changes you make to a child of the constructor function will not affect the constructor function, just like any changes you make to an object of a certain class won't afffect the overall class.
